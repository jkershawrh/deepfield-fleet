Common questions about fleet-llm-d and deepfield-fleet.

== "Is this a model server?"

No.
fleet-llm-d does not serve models.
It orchestrates the fleet of model servers -- OVMS, vLLM, TGI, or any OpenAI-compatible backend.

fleet-llm-d decides *where* models run (PlacementPolicy), *how* traffic routes (FleetRoutingPolicy), *when* to scale (FleetScalingPolicy), *who* gets what quota (TenantProfile), and records every decision in the ARE Immutable Ledger.

The model servers do the inference.
fleet-llm-d makes sure the right model serves the right request on the right hardware at the right cost.

== "Why not just use HPA?"

HPA is reactive -- it scales after the spike hits.
By the time new replicas are ready, users have already experienced latency breaches and 503 errors.

fleet-llm-d adds predictive scaling on top of HPA:

* *Event profiles*: YAML files that define scheduled events (e.g., Summit Connect, product launch). The `event_calendar` nanoagent pre-warms replicas before the event starts.
* *SLO forecasting*: Linear regression on P95 latency trends predicts breaches before they happen. "P95 will breach 5s SLO in 22 minutes."
* *Intent-driven scaling*: The predictive brain emits typed intents (PreWarm, Scale, ShedLoad) that fleet-llm-d evaluates against policy before executing.

HPA is still there as the safety net.
fleet-llm-d's brain acts before HPA needs to.

== "What about multi-cluster?"

Multi-cluster is fleet-llm-d's core design, not an add-on.

* *Rust gateway*: Routes inference requests across clusters with latency-aware load balancing
* *PlacementPolicy*: Spreads models geographically based on regulatory constraints and cost
* *KVCacheTransferPolicy*: Migrates hot KV caches between clusters for session continuity during failover
* *Fleet controller*: Federates metrics across all registered clusters for fleet-wide visibility
* *Cross-cluster routing*: The data plane handles HTTP/2 multiplexed connections with connection pooling

== "What if the brain is wrong?"

The policy evaluator is the safety gate between prediction and action.
Every intent from the predictive brain is evaluated before execution:

* *Confidence gate*: Intent confidence below threshold? Deferred. No action taken.
* *Replica limits*: Requested replicas exceed maximum? Refused. Capped at limit.
* *Budget constraints*: Action would exceed tenant budget? Refused.
* *Human approval*: Severity score above threshold? Requires human sign-off before executing.

And every prediction-action-outcome chain is recorded in the ARE Ledger.
Over time, you can measure prediction accuracy by comparing forecasts to actual outcomes.

== "What is the ARE Ledger?"

The ARE (Audit, Replay, Evidence) Immutable Ledger is independent shared infrastructure -- not a fleet-llm-d component.

It provides a tamper-proof audit trail using 5 chain types:

* *Placement* -- where models were placed and why
* *Scaling* -- when replicas changed and what triggered it
* *Routing* -- how traffic weights shifted
* *Lifecycle* -- canary rollout decisions and SLO gate results
* *Tenant* -- quota changes, rate limit adjustments, budget alerts

Each entry contains a SHA-256 hash of the previous entry in its chain.
Tamper with one entry, the chain breaks.
Proof receipts are portable -- auditors can verify independently.

fleet-llm-d is one of many writers.
Other platforms (MaaS, RHACM, agentic frameworks, CI/CD) can write to the same ledger instance.

== "How do I add a new model?"

Create a FleetInferencePool CRD:

[source,bash,role=execute]
----
cat <<'EOF' | oc apply -f -
apiVersion: fleet.llm-d.io/v1alpha1
kind: FleetInferencePool
metadata:
  name: my-model
  namespace: fleet-llm-d
spec:
  modelName: my-model
  modelVersion: "1.0"
  replicas: 1
  hardware:
    type: cpu
    accelerator: amx
  serving:
    runtime: ovms
    quantization: int8
    format: openvino-ir
EOF
----

The fleet controller reconciles the CRD and deploys the model to the best cluster based on PlacementPolicy.

== "How long to deploy?"

* *Helm install*: Under 2 minutes for fleet-llm-d on an existing cluster
* *First health check*: 30 seconds after pods are running
* *deepfield-fleet dashboard*: 1 minute (single container)
* *First inference*: Under 5 minutes from `helm install` to chat completion
* *Full production setup*: 15-30 minutes (Helm + backends + Grafana + ledger)

== "Can this run air-gapped?"

Yes.

* *fleet-llm-d*: Runs entirely within the cluster. No external API calls. The Go control plane and Rust data plane are self-contained.
* *Model servers*: OVMS and vLLM run locally. Models are loaded from local storage or an internal registry.
* *deepfield-fleet*: Runs as a local container. Connects to fleet-llm-d within the cluster network.
* *ARE Ledger*: Runs on its own infrastructure within your network.

The only external dependency is pulling container images, which can be mirrored to an internal registry.
