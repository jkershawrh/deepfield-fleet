== Three-Layer Architecture

The platform is built in three independently deployable layers: the fleet-llm-d actuator, the deepfield-fleet predictive brain, and the ARE Immutable Ledger.

== Layer 1: fleet-llm-d (Actuator)

fleet-llm-d is the orchestration core, built in Go (control plane) and Rust (data plane).

=== Go Control Plane

The control plane runs as a Kubernetes operator and exposes 31 API endpoints:

[cols="2,4"]
|===
| Subsystem | Capabilities

| *Fleet Management*
| Cluster registration, model deployment, health monitoring, fleet-wide status

| *Inference Proxy*
| OpenAI-compatible `/v1/chat/completions`, connection pooling, load shedding, SSE streaming

| *Placement Engine*
| GPU/CPU affinity, regulatory constraints, cost optimization, ModelPack OCI resolution

| *Scaling Engine*
| SLO-gated scaling, predictive pre-warming, intent evaluation, HPA coordination

| *Routing Engine*
| Weighted routing, geographic routing, failover, cross-cluster traffic management

| *Cost Engine*
| GPU/CPU chargeback, tier comparison (H200, B200, H100, A100, MI300X, L40, Xeon), savings calculation

| *Governance*
| Multi-tenant quotas, rate limiting, budget alerts, fairness scheduling
|===

=== 7 CRDs

[cols="2,4"]
|===
| CRD | Purpose

| `FleetInferencePool`
| Defines what to serve -- model name, version, replica count, hardware requirements

| `PlacementPolicy`
| Defines where models run -- regulatory constraints, hardware affinity, cost optimization

| `FleetRoutingPolicy`
| Defines how traffic flows -- weighted, geographic, failover routing

| `FleetScalingPolicy`
| Defines when to scale -- GPU utilization, latency, queue depth thresholds

| `TenantProfile`
| Defines who gets what -- token quotas, GPU budgets, rate limits per tenant

| `KVCacheTransferPolicy`
| Handles cross-cluster KV cache migration for session continuity

| `ModelLifecycle`
| Manages canary rollouts with SLO gates and automatic rollback
|===

=== Rust Data Plane

The Rust data plane handles the hot path:

* *Cross-cluster gateway* -- routes inference requests across clusters using tokio async runtime
* *Load balancing* -- weighted round-robin, least-connections, latency-aware routing
* *Connection pooling* -- reuses HTTP/2 connections to model servers
* *Load shedding* -- drops requests under overload with backpressure signals
* *SSE streaming* -- proxies Server-Sent Events for streaming inference responses
* *KV cache transfer* -- moves hot KV caches between clusters via gRPC

== Layer 2: deepfield-fleet (Predictive Brain)

deepfield-fleet is the predictive intelligence layer that sits on top of fleet-llm-d.
It predicts problems before they happen and emits typed intents to the actuator.

=== Fleet Nanoagents (4)

[cols="2,4"]
|===
| Agent | What It Does

| `slo_drift`
| Monitors P95 latency trends across the fleet, detects upward drift before SLO breach

| `capacity_pressure`
| Watches GPU/CPU utilization and queue depth across clusters

| `queue_depth`
| Tracks inference request queue length, detects backlog buildup

| `event_calendar`
| Reads YAML event profiles (e.g., summit_connect.yaml) and fires pre-warming before scheduled events
|===

=== SLO Forecaster (Microagent)

Runs linear regression on P95 latency time series.
Calculates time-to-breach and confidence interval.
Example: "At +80ms/min, P95 breaches 5000ms SLO in 22 minutes (confidence: 0.82)."

=== Consequence Scoper (Macroagent)

Calculates blast radius of a predicted breach:

....
severity_score = affected_models * affected_users * severity_weight
....

When the score crosses a threshold, the system requires human approval before acting.

=== Intent Emitter

Emits typed intents to fleet-llm-d:

[cols="2,4"]
|===
| Intent Type | Action

| `PreWarmIntent`
| Scale model replicas before a predicted event

| `ScaleIntent`
| Adjust replica count based on capacity pressure

| `ShedLoadIntent`
| Activate load shedding when overload is imminent

| `AlertIntent`
| Notify operators when severity exceeds human-gate threshold
|===

fleet-llm-d evaluates each intent against policy (confidence gates, replica limits, budget constraints) before executing.

== Layer 3: ARE Immutable Ledger

The ARE Ledger is independent shared infrastructure -- not a fleet-llm-d component.
It provides a tamper-proof audit trail for every fleet decision.

=== 5 Chain Types

[cols="2,4"]
|===
| Chain | What It Records

| `placement`
| Model placement decisions -- which cluster, why, cost delta

| `scaling`
| Scaling events -- replica changes, trigger reason, SLO state

| `routing`
| Routing changes -- traffic weights, failover events, geographic shifts

| `lifecycle`
| Model lifecycle events -- canary rollouts, promotions, rollbacks

| `tenant`
| Tenant governance events -- quota changes, rate limit adjustments, budget alerts
|===

=== Hash Linking

Each ledger entry contains a SHA-256 hash of the previous entry in its chain.
Tampering with any entry breaks the chain.
Proof receipts are portable and can be verified independently.

== Inference Proxy

The inference proxy provides an OpenAI-compatible interface:

* *Endpoint*: `/v1/chat/completions` -- drop-in replacement for OpenAI API
* *Connection pooling*: HTTP/2 multiplexed connections to backend model servers
* *Load shedding*: Returns 503 with `Retry-After` header under overload
* *SSE streaming*: Proxies `text/event-stream` responses for streaming inference
* *Routing headers*: `X-Fleet-Routed-To` (cluster), `X-Fleet-Routing-Reason` (why)
* *Fairness*: `x-llm-d-inference-fairness-id` header for tenant-aware scheduling

== CPU Inference Stack

The CPU inference path uses Intel Xeon 6 Granite Rapids:

[cols="1,3"]
|===
| Component | Detail

| *Hardware*
| Intel Xeon 6 Granite Rapids, 256 cores, AMX acceleration

| *Model Server*
| OVMS (OpenVINO Model Server) C++ runtime

| *Quantization*
| INT8 via Neural Compressor, leveraging AMX instructions

| *Models*
| Granite family (350M, 1B, 3B, 8B) in OpenVINO IR format

| *Cost*
| $0.60/hr vs $32/hr GPU (H100) -- 53x savings
|===
