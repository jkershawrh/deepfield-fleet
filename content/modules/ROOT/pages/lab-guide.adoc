= fleet-llm-d Hands-On Lab
:toc: left
:toclevels: 3

Self-paced lab for learning fleet-llm-d inference orchestration.
5 modules, approximately 60 minutes.

== Prerequisites

* An OpenShift sandbox with fleet-llm-d and synthetic-maas pre-deployed
* Terminal access (via showroom or local)
* A modern web browser (Chrome, Firefox, Edge)
* The following variables are provided by your lab environment:
** `{fleet_url}` -- fleet-llm-d API endpoint
** `{fleet_token}` -- authentication token
** `{openshift_api_url}` -- OpenShift API URL
** `{user}` / `{password}` -- OpenShift credentials

== Module 1: Deploy fleet-llm-d (15 min)

=== 1.1 Inspect the Helm Chart

Start by examining the fleet-llm-d Helm chart to understand the deployment configuration.

[source,bash,role=execute]
----
cat charts/fleet-llm-d/values.yaml
----

Review the key sections: `mode`, `auth`, `rateLimit`, `loadShedding`, and `backends`.

=== 1.2 Deploy fleet-llm-d

Install fleet-llm-d using the Helm chart with authentication enabled:

[source,bash,role=execute]
----
helm install fleet-llm-d charts/fleet-llm-d/ \
  --set mode=all \
  --set auth.enabled=true \
  --set auth.secretName=fleet-auth
----

The `mode=all` flag deploys the fleet controller, inference proxy, and cost engine together.

=== 1.3 Verify the Deployment

Check that fleet-llm-d is healthy:

[source,bash,role=execute]
----
curl {fleet_url}/healthz
----

Expected response: `{"status":"ok"}`.

=== 1.4 Register a Cluster

Register your lab cluster with fleet-llm-d:

[source,bash,role=execute]
----
curl -X POST {fleet_url}/api/v1/clusters \
  -H "Authorization: Bearer {fleet_token}" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "lab-cluster",
    "region": "us-east",
    "hardware": "intel-xeon-6",
    "capabilities": ["cpu-inference", "amx"]
  }'
----

=== 1.5 Explore the CRDs

List the fleet-llm-d CRDs installed on your cluster:

[source,bash,role=execute]
----
oc get crd | grep fleet
----

You should see 7 CRDs:

* `fleetinferencepools.fleet.llm-d.io`
* `placementpolicies.fleet.llm-d.io`
* `fleetroutingpolicies.fleet.llm-d.io`
* `fleetscalingpolicies.fleet.llm-d.io`
* `tenantprofiles.fleet.llm-d.io`
* `kvcachetransferpolicies.fleet.llm-d.io`
* `modellifecycles.fleet.llm-d.io`

=== Checkpoint

[%interactive]
* [ ] `curl {fleet_url}/healthz` returns 200
* [ ] Cluster registered successfully
* [ ] 7 CRDs visible in `oc get crd`

== Module 2: Serve a Model on CPU (10 min)

Your sandbox includes a pre-deployed **synthetic-maas** emulator with 19 models — the same API contract as production MaaS, zero GPU required. fleet-llm-d routes inference through it.

=== 2.1 Create a FleetInferencePool

Register a Granite 350M model by creating a FleetInferencePool CRD. The inference backend (synthetic-maas) is already running:

[source,bash,role=execute]
----
cat <<'EOF' | oc apply -f -
apiVersion: fleet.llm-d.io/v1alpha1
kind: FleetInferencePool
metadata:
  name: granite-350m
  namespace: fleet-llm-d
spec:
  modelName: granite-350m
  modelVersion: "1.0"
  replicas: 1
  hardware:
    type: cpu
    accelerator: amx
  serving:
    runtime: ovms
    quantization: int8
    format: openvino-ir
  resources:
    requests:
      cpu: "4"
      memory: "8Gi"
    limits:
      cpu: "8"
      memory: "16Gi"
EOF
----

Verify the pool is ready:

[source,bash,role=execute]
----
oc get fleetinferencepool granite-350m -n fleet-llm-d
----

=== 2.2 Send an Inference Request

Send a chat completion request through the fleet-llm-d proxy:

[source,bash,role=execute]
----
curl -s {fleet_url}/v1/chat/completions \
  -H "Authorization: Bearer {fleet_token}" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "granite-350m",
    "messages": [
      {"role": "user", "content": "Hello, what can you help me with?"}
    ],
    "max_tokens": 50
  }' | jq .
----

=== 2.3 Check Routing Headers

Send a request with verbose headers to see where it was routed:

[source,bash,role=execute]
----
curl -sv {fleet_url}/v1/chat/completions \
  -H "Authorization: Bearer {fleet_token}" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "granite-350m",
    "messages": [{"role": "user", "content": "Hello"}],
    "max_tokens": 10
  }' 2>&1 | grep -i "x-fleet"
----

Look for:

* `X-Fleet-Routed-To` -- the cluster that served the request
* `X-Fleet-Routing-Reason` -- why that cluster was chosen (e.g., "least-latency", "cost-optimal")

=== 2.4 View Cost Savings

Check the cost comparison between GPU and CPU inference:

[source,bash,role=execute]
----
curl -s "{fleet_url}/api/v1/cost/savings?model=granite-350m&duration=1h" \
  -H "Authorization: Bearer {fleet_token}" | jq .
----

=== Checkpoint

[%interactive]
* [ ] FleetInferencePool `granite-350m` shows `Ready`
* [ ] Chat completion returns a response
* [ ] `X-Fleet-Routed-To` header present in response
* [ ] Cost savings API returns GPU vs CPU comparison

== Module 3: Multi-Tenant Governance (10 min)

=== 3.1 Create a TenantProfile

Define a tenant with token quotas and rate limits:

[source,bash,role=execute]
----
cat <<'EOF' | oc apply -f -
apiVersion: fleet.llm-d.io/v1alpha1
kind: TenantProfile
metadata:
  name: tenant-a
  namespace: fleet-llm-d
spec:
  displayName: "Team Alpha"
  quotas:
    tokensPerMinute: 1000
    tokensPerDay: 100000
    maxConcurrent: 5
  rateLimit:
    requestsPerMinute: 10
    burstSize: 15
  budget:
    dailyLimit: "50.00"
    alertThreshold: "40.00"
    currency: "USD"
  allowedModels:
    - "granite-350m"
    - "granite-3b"
EOF
----

=== 3.2 Send a Request with Fairness Header

Send an inference request tagged with the tenant's fairness ID:

[source,bash,role=execute]
----
curl -s {fleet_url}/v1/chat/completions \
  -H "Authorization: Bearer {fleet_token}" \
  -H "x-llm-d-inference-fairness-id: tenant-a" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "granite-350m",
    "messages": [{"role": "user", "content": "Summarize the benefits of CPU inference."}],
    "max_tokens": 100
  }' | jq .
----

=== 3.3 Exceed the Rate Limit

Send rapid requests to trigger the rate limit (10 requests/minute):

[source,bash,role=execute]
----
for i in $(seq 1 12); do
  echo "Request $i:"
  curl -s -o /dev/null -w "%{http_code}" \
    {fleet_url}/v1/chat/completions \
    -H "Authorization: Bearer {fleet_token}" \
    -H "x-llm-d-inference-fairness-id: tenant-a" \
    -H "Content-Type: application/json" \
    -d '{"model":"granite-350m","messages":[{"role":"user","content":"test"}],"max_tokens":5}'
  echo ""
done
----

After 10 requests, you should see HTTP 429 (Too Many Requests) responses.

=== 3.4 Check Budget Alerts

View budget alert status for the tenant:

[source,bash,role=execute]
----
curl -s {fleet_url}/api/v1/cost/alerts \
  -H "Authorization: Bearer {fleet_token}" | jq .
----

=== Checkpoint

[%interactive]
* [ ] TenantProfile `tenant-a` created
* [ ] Requests with `x-llm-d-inference-fairness-id` header succeed
* [ ] Rate limit triggers 429 after exceeding threshold
* [ ] Budget alerts API returns tenant cost data

If you want to drive the operations manually instead of using the walkthrough:

Load an event profile:

[source,bash,role=execute]
----
curl -X POST {fleet_url}/api/v1/events/load \
  -H "Authorization: Bearer {fleet_token}" \
  -H "Content-Type: application/json" \
  -d '{"profile": "summit_connect"}'
----

Run the SLO forecast:

[source,bash,role=execute]
----
curl -s {fleet_url}/api/v1/forecast/slo \
  -H "Authorization: Bearer {fleet_token}" | jq .

Scope the blast radius:

[source,bash,role=execute]
----
curl -s {fleet_url}/api/v1/forecast/blast-radius \
  -H "Authorization: Bearer {fleet_token}" | jq .

Emit a pre-warm intent:

[source,bash,role=execute]
----
curl -X POST {fleet_url}/api/v1/intents \

    "model": "granite-4.1-3b",

== Module 5: Observe and Prove (10 min)

=== 5.1 Open Grafana

Open the fleet Grafana dashboard:

....
https://fleet-grafana.{openshift_api_url}/
....

Log in with `{user}` / `{password}`.

=== 5.2 View the Fleet Overview Dashboard

Navigate to *Dashboards -> Fleet Overview*. The synthetic-maas emulator produces realistic vLLM Prometheus metrics — the dashboards show real-looking data:

* *Cluster Health* -- status of all registered clusters
* *Model Serving Metrics* -- request rate, latency P50/P95/P99, error rate per model (from synthetic-maas vLLM metrics)
* *KV Cache* -- cache hit rate, prefix reuse (the emulator simulates a real prefix cache)
* *Scaling Events* -- replica timeline, intent history, SLO compliance
* *Governance* -- tenant quota utilization, rate limit triggers, budget alerts

=== 5.3 Run the Smoke Test Harness

Run the fleet test harness to validate end-to-end functionality:

[source,bash,role=execute]
----
oc run fleet-harness \
  --image=quay.io/deepfield-fleet/fleet-harness:latest \
  --restart=Never \
  --env="FLEET_URL={fleet_url}" \
  --env="FLEET_TOKEN={fleet_token}" \
  -- --suite=smoke
----

Watch the test output:

[source,bash,role=execute]
----
oc logs -f fleet-harness
----

The smoke suite verifies:

* Health endpoint responds
* Inference proxy serves completions
* Tenant governance enforces quotas
* Cost engine calculates savings
* Ledger chains are valid

=== 5.4 Verify ARE Ledger Chains

Run a final verification of all ledger chains:

[source,bash,role=execute]
----
curl -s {fleet_url}/api/v1/verify/chains \
  -H "Authorization: Bearer {fleet_token}" | jq .
----

Each chain should show:

* `chain_type` -- placement, scaling, routing, lifecycle, or tenant
* `entries` -- count of recorded decisions
* `valid` -- true (hash chain intact)
* `last_hash` -- SHA-256 hash of the latest entry

=== 5.5 Clean Up

Remove the test pod:

[source,bash,role=execute]
----
oc delete pod fleet-harness
----
