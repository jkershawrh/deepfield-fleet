== The Problem: GPU Scarcity and Cluster Sprawl

Modern enterprises face four compounding challenges with AI inference:

*GPU scarcity.* H100 and H200 GPUs have multi-month lead times.
Teams wait weeks for allocation.
When hardware arrives, it is siloed within a single cluster.

*No multi-cluster orchestration.* Each cluster runs its own model servers independently.
There is no fleet-wide view of capacity, no cross-cluster routing, no coordinated placement.
The same model may be deployed redundantly across clusters with no load balancing.

*Reactive scaling.* Kubernetes HPA scales after the spike hits.
By the time replicas come online, users have already experienced latency breaches and 503 errors.
There is no way to pre-warm for known events.

*No governance.* There is no multi-tenant fairness, no cost chargeback, no audit trail.
Every team runs their own inference with no visibility into who is consuming what at what cost.

== The Solution: fleet-llm-d

fleet-llm-d is a fleet-level inference orchestration platform built on llm-d.
It does not serve models -- it orchestrates the fleet of model servers.

=== 7 CRDs

The fleet's desired state is defined declaratively through 7 Kubernetes CRDs:

* *FleetInferencePool* -- what to serve (model, version, replicas, hardware)
* *PlacementPolicy* -- where to run (regulatory constraints, hardware affinity, cost optimization)
* *FleetRoutingPolicy* -- how traffic flows (weighted, geographic, failover)
* *FleetScalingPolicy* -- when to scale (GPU utilization, latency, queue depth)
* *TenantProfile* -- who gets what (token quotas, GPU budgets, rate limits)
* *KVCacheTransferPolicy* -- cross-cluster KV cache migration
* *ModelLifecycle* -- canary rollouts with SLO gates and automatic rollback

=== Control Plane + Data Plane

The Go control plane exposes 31 API endpoints covering fleet management, inference proxying, placement, scaling, routing, cost/chargeback, and governance.
It runs as a Kubernetes operator that reconciles CRDs continuously.

The Rust data plane handles cross-cluster traffic routing, load balancing, connection pooling, load shedding, and SSE streaming.
Built on tokio, tonic, and axum, it handles the hot path with minimal latency overhead.

Intel Xeon 6 Granite Rapids processors provide 256 cores with AMX (Advanced Matrix Extensions) -- the same matrix acceleration that GPUs use, built into the CPU.
OVMS (OpenVINO Model Server) serves INT8 quantized models in C++.
Neural Compressor handles the FP32-to-INT8 quantization.

The result: Granite models (350M through 8B parameters) serve on CPU at P95 latency under 5 seconds, at 53x lower cost than GPU.
This is not a theoretical claim -- it is measured on production clusters with OVMS C++.

For workloads that require GPU (large models, high throughput, low-latency requirements), fleet-llm-d manages GPU clusters alongside CPU clusters.
The cost engine calculates chargeback across H200, B200, H100, A100, MI300X, L40, and Xeon CPU.

== Predictive Intelligence: deepfield-fleet

On top of the fleet-llm-d actuator sits deepfield-fleet -- a predictive intelligence layer.

=== Four Fleet Nanoagents

* *slo_drift* -- monitors P95 latency trends, detects upward drift before SLO breach
* *capacity_pressure* -- watches GPU/CPU utilization and queue depth
* *queue_depth* -- tracks inference request queue length, detects backlog buildup
* *event_calendar* -- reads YAML event profiles and fires pre-warming before scheduled events

=== SLO Forecaster

A microagent runs linear regression on P95 latency time series.
It calculates time-to-breach and confidence interval.
Example output: "At +80ms/min, P95 breaches the 5000ms SLO in 22 minutes (confidence: 0.82)."

=== Consequence Scoper

A macroagent calculates blast radius: affected models times affected users times severity weight.
When the severity score crosses a threshold, the system requires human approval before acting.

=== Intent-Driven Scaling

The brain emits typed intents to fleet-llm-d: PreWarm, Scale, ShedLoad, Alert.
fleet-llm-d evaluates each intent against policy -- confidence gates, replica limits, budget constraints -- before executing.
This is predictive scaling with safety gates, not autonomous scaling without guardrails.

== ARE Immutable Ledger

The ARE Ledger is independent shared infrastructure -- not a fleet-llm-d component.
fleet-llm-d is one of many writers; other platforms (MaaS, RHACM, agentic frameworks) can write to the same instance.

=== 5 Chain Types

* *Placement* -- model placement decisions (which cluster, why, cost delta)
* *Scaling* -- replica changes (trigger reason, SLO state, intent ID)
* *Routing* -- traffic weight changes (failover events, geographic shifts)
* *Lifecycle* -- canary rollouts (promotions, rollbacks, SLO gate results)
* *Tenant* -- governance events (quota changes, rate limit adjustments, budget alerts)

=== Hash Linking

Each entry contains a SHA-256 hash of the previous entry in its chain.
Tampering with any entry breaks the chain.
Proof receipts are portable and can be verified independently by auditors.

=== Compliance Value

For regulated industries (finance, healthcare, government), the ledger provides:

* Tamper-proof evidence of every infrastructure decision
* Chain verification independent of fleet-llm-d
* Portable proof receipts for external auditors
* Time-stamped record of who requested what, what policy was evaluated, and what action was taken
